# PRD — Приложение «Контроль заявок»

## Обзор продукта

«Контроль заявок» — Android-приложение для частного мастера по ремонту холодильного оборудования. Приложение предназначено для личного использования одним человеком. Это не CRM, не корпоративный сервис и не инструмент для команды.

Приложение решает одну конкретную проблему: мастеру постоянно звонят клиенты, просят перезвонить завтра или послезавтра, приехать в определённый день, уточнить детали позже. Без системы такие договорённости держатся в голове, в заметках и переписках — и часть из них теряется. Приложение заменяет этот хаос простым списком заявок, отсортированным по времени ближайшего действия.

Главный принцип продукта — скорость и простота. Мастер должен создать заявку за 20–40 секунд сразу после звонка, одной рукой, без обучения и инструкций. Поэтому обязательных полей всего три, интерфейс светлый и спокойный, лишних функций нет. Весь рабочий цикл умещается в одну фразу: создал заявку → увидел в списке → выполнил или отменил → заявка ушла в историю.

Приложение работает полностью офлайн. Все данные хранятся локально на телефоне, сохраняются после закрытия приложения и перезагрузки устройства. Регистрация, аккаунт, сервер и интернет не требуются.

## Целевой пользователь

Один частный мастер, который принимает заявки по телефону и выезжает на ремонт. Работает с приложением в движении, часто одной рукой, между звонками и выездами. Ценит, чтобы приложение не отвлекало и не требовало лишних действий.

## Цели продукта

- Не терять клиентов, которым нужно перезвонить или приехать позже.
- Быстро фиксировать заявку сразу после звонка.
- Видеть ближайшие задачи в понятном хронологическом списке.
- Чётко отделять активные заявки от завершённых.
- Фиксировать результат каждой заявки: выполнена или сорвалась.
- Хранить простую историю выполненных и отменённых работ.
- Не перегружать пользователя лишними функциями.
- Работать без интернета и без какой-либо настройки.

## Границы продукта (версия 1)

Приложение намеренно остаётся минимальным. В первой версии НЕ реализуются: регистрация и личный кабинет, облачная синхронизация, карты и маршруты, склад запчастей, бухгалтерия, аналитика, отчёты и графики, экспорт в Excel, роли и несколько сотрудников, интеграции с мессенджерами, приём оплаты, воронки продаж, фотографии, перегруженная админ-панель. Всё это может быть добавлено позже, если приложение окажется полезным на практике.

Структурно приложение состоит только из трёх вкладок нижнего меню. Никаких дополнительных разделов в первой версии нет.

## Фичи

### 1. Навигация по трём вкладкам

В нижней части экрана расположено постоянное меню из трёх вкладок: **Создание**, **Список**, **История**. Пользователь переключается между ними одним касанием. Это единственная навигация в приложении — никаких боковых меню, вложенных разделов и скрытых экранов. «Список» — главный рабочий экран, на нём пользователь проводит большую часть времени.

### 2. Создание заявки

Вкладка «Создание» предназначена для быстрого добавления новой заявки сразу после звонка или сообщения клиента.

Форма содержит три обязательных поля:
- **Название заявки** — короткое описание сути (например, «Samsung No Frost не морозит», «Витрина в магазине», «Ледогенератор ошибка», «Холодильник Атлант течёт»).
- **Телефон клиента** — ввод в свободном формате, поддерживаются записи вида `+7…`, `8…`, с пробелами, скобками и дефисами.
- **Дата и время следующего действия** — момент, когда нужно совершить следующий шаг (позвонить, приехать, уточнить, напомнить, проверить актуальность, договориться о времени). Поле называется именно «дата следующего действия», а не «дедлайн».

Форма также содержит необязательные поля:
- **Имя клиента** — текстовый ввод вручную.
- **Адрес** — обычное текстовое поле.
- **Тип оборудования** — выпадающий список: бытовой холодильник, торговое оборудование, промышленное оборудование, ледогенератор, морозильная камера, холодильная витрина, сплит-система / кондиционер, другое.
- **Тип действия** — выпадающий список: позвонить, приехать, диагностика, ремонт, ожидание детали, уточнить информацию, другое.
- **Комментарий** — свободный текст (например: «Клиент говорит, что морозилка работает, а холодильная камера нет. Позвонить завтра после 12:00.»).

На экране есть кнопка **«Сохранить заявку»** и кнопка **«Очистить форму»**. Опционально может присутствовать кнопка **«Выбрать контакт»** (см. фичу 12).

При сохранении приложение проверяет, что заполнены все три обязательных поля. Если хотя бы одно из них пустое, показывается ошибка с текстом: «Заполните название, телефон и дату следующего действия.» — и заявка не сохраняется. Если все обязательные поля заполнены, заявка сохраняется со статусом «Активная», показывается сообщение «Заявка создана», форма очищается, и пользователь автоматически переходит на вкладку «Список», где сразу видит новую заявку.

### 3. Список активных заявок

Вкладка «Список» показывает все активные заявки, которые ещё нужно обработать. Это главный экран приложения.

Заявки всегда отсортированы по дате следующего действия по возрастанию — ближайшие сверху. За счёт этого просроченные заявки (их дата уже прошла) оказываются в самом верху, затем сегодняшние, завтрашние и более дальние. Самая срочная заявка всегда наверху списка.

Каждая заявка отображается отдельной карточкой. В карточке показываются: название заявки, имя клиента или телефон, дата и время следующего действия, тип оборудования, тип действия и короткий адрес (если указан). Пример карточки:

> **Samsung No Frost не морозит**
> Иван / +7 999 123-45-67
> Сегодня, 16:00 — позвонить
> Бытовой холодильник
> м. Отрадное

Интерфейс спокойный, без кислотных цветов. Используется простая цветовая индикация срочности: красный маркер — заявка просрочена, оранжевый — на сегодня, нейтральный цвет — будущая заявка. Сложная легенда и множество цветов не используются.

### 4. Детальная карточка заявки

При нажатии на карточку в списке открывается подробный просмотр заявки — модальное нижнее окно (BottomSheet) или отдельный экран. В нём показаны все данные заявки и доступны кнопки действий: **Позвонить**, **Изменить**, **Перенести**, **Выполнено**, **Отменить**, **Закрыть**. «Изменить» позволяет отредактировать поля заявки, «Перенести» — изменить дату следующего действия, остальные кнопки выполняют соответствующие действия над заявкой.

### 5. Завершение заявки

Пользователь отмечает заявку выполненной двумя способами: свайпом карточки вправо в списке или кнопкой «Выполнено» в детальной карточке. Во время свайпа вправо может показываться иконка галочки. Заявка получает статус «Выполнена», в неё записывается дата завершения, она исчезает из активного списка и появляется во вкладке «История». Позже к ней можно добавить итоговую стоимость и итоговый комментарий.

### 6. Отмена заявки

Пользователь отменяет заявку свайпом карточки влево или кнопкой «Отменить» в детальной карточке. Во время свайпа влево может показываться иконка крестика или корзины. При отмене приложение предлагает указать причину — выбрать из готовых вариантов или написать свою. Готовые причины: клиент не отвечает, клиент отказался, клиент выбрал другого мастера, заявка потеряла актуальность, слишком далеко, не договорились по цене, другое.

После отмены заявка получает статус «Отменена», в неё записывается дата отмены, она исчезает из активного списка и появляется в «Истории». Заявка не удаляется из базы физически даже если клиент «слился» — она сохраняется в истории со статусом «Отменена», потому что это полезнее полного удаления. В интерфейсе основное действие называется «Отменить», а не «Удалить».

### 7. Защита от случайного свайпа

Свайпы намеренно лёгкие, поэтому после каждого свайпа (вправо или влево) приложение показывает короткое уведомление (Snackbar) с текстом «Заявка перенесена в историю» и кнопкой «Отменить». Если пользователь нажимает «Отменить», заявка возвращается в активный список в прежнем состоянии. Это защищает от потери заявки из-за случайного жеста.

### 8. История

Вкладка «История» показывает все заявки, которые больше не активны, — выполненные и отменённые. Каждая отображается карточкой. В карточке истории показываются: название заявки, клиент/телефон, дата создания, дата завершения или отмены, статус, итоговая стоимость (если указана) и краткий итоговый комментарий. Примеры:

> **Витрина в магазине не холодит**
> Статус: выполнена
> Стоимость: 6500 ₽
> Итог: заменён вентилятор обдува.

> **Ледогенератор ошибка**
> Статус: отменена
> Причина: клиент выбрал другого мастера.

В истории используется цветовая индикация статуса: зелёный — выполнено, серый — отменено.

На экране истории есть переключатель сортировки с двумя режимами. **По хронологии** — сначала последние завершённые или отменённые заявки (по дате закрытия, новые сверху). **По статусу** — заявки сгруппированы: сначала выполненные, затем отменённые. Переключатель оформлен как простая кнопка «Сортировка: по дате / по статусу», меняющая режим по нажатию.

Дополнительно доступен минимальный фильтр: все / выполненные / отменённые.

### 9. Итоги заявки в истории

Пользователь может открыть любую заявку из истории и дополнить её итоговой информацией. Для выполненной заявки добавляются итоговая стоимость ремонта и итоговый комментарий о том, что было сделано (например: «Почистил конденсатор, заменил пусковое реле. Клиент оплатил 4500 ₽.» или «Диагностика показала утечку, клиент пока думает.»). Для отменённой заявки добавляются причина отмены и комментарий (например: «Клиент не отвечает два дня.», «Сказал, что уже вызвал другого мастера.», «Не устроила цена.»). Эти поля можно заполнить и изменить в любой момент после переноса заявки в историю.

### 10. Звонок клиенту

В детальной карточке заявки есть кнопка «Позвонить». При нажатии открывается стандартная звонилка телефона с уже подставленным номером клиента. Приложение не совершает звонок автоматически — пользователь сам подтверждает вызов в системной звонилке.

### 11. Напоминания

Если у заявки указаны дата и время следующего действия, приложение отправляет локальное уведомление в это время. Пример уведомления:

> Позвонить: Samsung No Frost не морозит
> Сегодня 15:00
> +7 999 123-45-67

Минимальное поведение: нажатие на уведомление открывает карточку соответствующей заявки. Опционально уведомление может содержать кнопки быстрых действий (позвонить, выполнено, перенести), но для первой версии достаточно открытия карточки.

### 12. Импорт контакта (опционально)

Желательная, но не обязательная для первой версии функция. На форме создания может быть кнопка «Выбрать контакт». При нажатии открывается системный выбор одного контакта, после чего в форму подставляются имя и телефон клиента. Приложение использует именно системный выбор одного контакта и не запрашивает доступ ко всей телефонной книге.

### 13. Локальное хранение

Все заявки хранятся в локальной базе данных на телефоне. Данные сохраняются после закрытия приложения и после перезагрузки устройства. Приложение полностью работает без интернета. Сервер, облако, регистрация и авторизация не используются.

## Модель данных

Приложение хранит заявки в одной таблице `requests`. Сущность `Request` содержит поля:

- `id` — уникальный идентификатор.
- `title` — название заявки.
- `clientName` — имя клиента.
- `phone` — телефон клиента.
- `address` — адрес.
- `equipmentType` — тип оборудования.
- `actionType` — тип действия.
- `nextActionDateTime` — дата и время следующего действия.
- `comment` — комментарий.
- `status` — статус заявки: `ACTIVE`, `COMPLETED` или `CANCELLED`.
- `finalPrice` — итоговая стоимость.
- `finalComment` — итоговый комментарий.
- `cancelReason` — причина отмены.
- `createdAt` — дата создания.
- `updatedAt` — дата последнего изменения.
- `closedAt` — дата выполнения или отмены.

Обязательные (непустые) поля: `id`, `title`, `phone`, `nextActionDateTime`, `status`, `createdAt`. Остальные поля могут быть пустыми.

Заявка имеет ровно три статуса: **Активная** (`ACTIVE`), **Выполнена** (`COMPLETED`), **Отменена** (`CANCELLED`). При создании заявка получает статус `ACTIVE`. При завершении статус становится `COMPLETED`, при отмене — `CANCELLED`; в обоих случаях в `closedAt` записывается текущие дата и время, и заявка перестаёт показываться в активном списке. Поле `updatedAt` обновляется при каждом изменении заявки.

Активные заявки сортируются по `nextActionDateTime` по возрастанию. История по умолчанию сортируется по `closedAt` по убыванию; при сортировке по статусу — сначала по статусу, затем по `closedAt` по убыванию.

## Рекомендуемый стек

Kotlin, Jetpack Compose, Room (SQLite), ViewModel, StateFlow/LiveData, Material Design 3, AlarmManager или WorkManager для уведомлений. Архитектура: UI → ViewModel → Repository → локальная база данных. Допустим другой стек при соблюдении обязательных требований: нативное Android-приложение, локальная база, три вкладки, простота, работа без интернета.

## Требования к интерфейсу

Интерфейс простой, спокойный, без лишних элементов и перегруженных меню, понятный без инструкции и рассчитанный на работу одной рукой сразу после звонка. Стиль: светлая тема, крупные кнопки, читаемый шрифт, карточки заявок, минимум цветов, нижнее меню из трёх вкладок. Ключевой ориентир: создание заявки занимает 20–40 секунд, поэтому обязательны только три поля (название, телефон, дата следующего действия), всё остальное — по желанию.

## Критерии готовности

Приложение считается готовым, когда выполняются все условия:

1. Пользователь может создать заявку.
2. Без названия, телефона и даты заявка не сохраняется.
3. Созданная заявка появляется во вкладке «Список».
4. Список отсортирован от ближайших заявок к дальним.
5. Пользователь может открыть заявку и посмотреть подробности.
6. Пользователь может позвонить клиенту из карточки заявки.
7. Пользователь может смахнуть заявку как выполненную.
8. Выполненная заявка исчезает из списка и появляется в истории.
9. Пользователь может смахнуть заявку как отменённую.
10. Отменённая заявка исчезает из списка и появляется в истории.
11. В истории можно отсортировать заявки по дате.
12. В истории можно отсортировать заявки по статусу.
13. К выполненной заявке можно добавить итоговую стоимость.
14. К выполненной заявке можно добавить итоговый комментарий.
15. К отменённой заявке можно добавить причину отмены.
16. Все данные сохраняются после закрытия приложения.
17. Приложение работает без интернета.
18. Интерфейс не перегружен и состоит только из трёх основных вкладок.
